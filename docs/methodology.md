# 提取方法

## 数据来源

当前数据来自样本 APK 解包后的路由注册代码目录。

提取对象是由应用构建产物生成的路由注册 smali，而不是手工搜索散落在业务代码中的字符串。

## 提取对象

主提取脚本会扫描每个 `JInterMediateRoute` 注册块，并记录：

- 该块内出现的所有 route pattern
- 当前块推断出的路由类型
- 路由所在文件与行号
- 同一块中的别名集合

## 为什么选择 `JInterMediateRoute`

原因有三个：

- 这是路由注册的集中落点，信息更完整。
- 同一块内能同时看到别名、类型、目标元信息，便于建立关联。
- 相比单纯全局搜字符串，误报更少，也更容易追溯来源。

## 输出文件

- `data/routes.json`：完整结构化结果
- `data/routes.csv`：便于筛选、透视和外部处理
- `data/summary.json`：聚合统计
- `docs/prefix-index.md` 与 `docs/prefixes/*.md`：从 `routes.json` 自动派生出的文档视图

## 可信度边界

以下结论可以认为是确定的：

- 某条路由 pattern 出现在生成路由表中
- 某些 pattern 属于同一注册块
- 某个注册块被标记为 `native`、`web` 或其他类型
- 某条路由在当前样本中的文件与行号位置

以下结论不能仅靠当前静态提取直接保证：

- 某条路由一定能被外部应用直接唤起
- 某个 query 参数一定是外部字符串传入，而不是路由层补写
- 某条路由在所有版本中都存在
- 某条路由最终一定到达某个页面且没有额外权限、登录或实验开关限制

## 已知局限

- 当前主提取目录集中在 `smali_classes14` 的 codegen 路由表。
- `unknown` 类型表示当前块内没有提取到明确类型字符串，不等于该路由无类型。
- 当前公开版索引不纳入 `http(s)` 规则与别名。
